# 第4课：外键 FOREIGN KEY 与参照完整性

---

## 一、什么是外键？

想象一个场景：
- 员工表中，张三的所属部门是 "D001"
- 但是部门表中，根本不存在 "D001" 这个部门！
- 这就出现了**数据不一致**的问题

**外键就是用来防止这种问题的**——它确保"引用的值在另一个表中必须存在"。

> **通俗理解**：外键就像一个"查岗机制"。你说你属于D001部门？让我去部门表里查查，D001到底存不存在！不存在就不让你填。

---

## 二、外键的基本语法

```sql
FOREIGN KEY(本表字段) REFERENCES 另一张表(那张表的字段)
```

以职位表为例：

```sql
FOREIGN KEY(dept_code) REFERENCES department(dept_code)
    ON DELETE RESTRICT
    ON UPDATE CASCADE
```

**解读**：
| 部分 | 含义 |
|------|------|
| `FOREIGN KEY(dept_code)` | 本表的 dept_code 是外键 |
| `REFERENCES department(dept_code)` | 它引用的是 department 表的 dept_code |
| `ON DELETE RESTRICT` | 被引用的部门被删除时，拒绝删除 |
| `ON UPDATE CASCADE` | 被引用的部门代码被修改时，跟着一起改 |

---

## 三、ON DELETE 和 ON UPDATE 的四种策略

这是外键最核心的部分——当被引用的数据发生变化时，怎么处理？

### 3.1 RESTRICT —— 拒绝（限制）

```
场景：你想删除部门D001，但职位表中还有属于D001的职位
结果：❌ 删除失败！必须先把相关职位删掉或改掉，才能删部门
```

> **类比**：你想拆掉一栋楼，但里面还有人住着，不允许拆。

### 3.2 CASCADE —— 级联（跟着变）

```
场景：部门D001改名为D001-NEW
结果：所有引用D001的记录自动跟着改成D001-NEW
```

```
场景：删除部门D001
结果：所有属于D001的记录也一起被删除！⚠️ 危险！
```

> **类比**：你搬家了，你的所有快递地址自动跟着改到新地址。

### 3.3 SET NULL —— 设为空

```
场景：删除员工1001（他是某部门的负责人）
结果：部门表中 dept_manager_emp 自动变成 NULL
```

> **类比**：班主任辞职了，班级的"班主任"一栏暂时空着，等新老师来再填。

### 3.4 NO ACTION —— 与 RESTRICT 类似

在MySQL中，`NO ACTION` 和 `RESTRICT` 效果一样，都是拒绝操作。

## 四、你项目中的外键策略分析

### 4.1 ON DELETE RESTRICT + ON UPDATE CASCADE（最常见）

```sql
-- 职位表引用部门表
FOREIGN KEY(dept_code) REFERENCES department(dept_code)
    ON DELETE RESTRICT ON UPDATE CASCADE

-- 员工表引用职位表
FOREIGN KEY(pos_id) REFERENCES position(pos_id)
    ON DELETE RESTRICT ON UPDATE CASCADE

-- 员工表引用部门表
FOREIGN KEY(dept_code) REFERENCES department(dept_code)
    ON DELETE RESTRICT ON UPDATE CASCADE
```

| 策略 | 含义 | 为什么这么选？ |
|------|------|---------------|
| ON DELETE RESTRICT | 不让删 | 有员工/职位的部门不能删，防止数据丢失 |
| ON UPDATE CASCADE | 跟着改 | 部门代码改了，相关记录自动更新，省事 |

> **这是最安全的组合**：删不行（怕丢数据），改可以（跟着变）。

### 4.2 ON DELETE CASCADE + ON UPDATE CASCADE

```sql
-- 支付方式表引用员工表
FOREIGN KEY(emp_id) REFERENCES employee(emp_id)
    ON DELETE CASCADE ON UPDATE CASCADE

-- 考勤表引用员工表
FOREIGN KEY(emp_id) REFERENCES employee(emp_id)
    ON DELETE CASCADE ON UPDATE CASCADE

-- 项目成员表引用员工表和项目表
FOREIGN KEY(emp_id) REFERENCES employee(emp_id)
    ON DELETE CASCADE ON UPDATE CASCADE
FOREIGN KEY(proj_id) REFERENCES project(proj_id)
    ON DELETE CASCADE ON UPDATE CASCADE
```

| 策略 | 含义 | 为什么这么选？ |
|------|------|---------------|
| ON DELETE CASCADE | 跟着删 | 员工离职了，他的考勤、加班记录也没必要保留 |

> **思考**：为什么员工删了，考勤记录也跟着删？因为考勤记录依附于员工存在，没有"无主"的考勤记录。

### 4.3 ON DELETE SET NULL + ON UPDATE CASCADE

```sql
-- 部门表引用员工表（负责人）
ALTER TABLE department ADD CONSTRAINT fk_dept_mgr
    FOREIGN KEY(dept_manager_emp) REFERENCES employee(emp_id)
    ON DELETE SET NULL ON UPDATE CASCADE
```

| 策略 | 含义 | 为什么这么选？ |
|------|------|---------------|
| ON DELETE SET NULL | 设为空 | 负责人离职了，部门还在，只是暂时没有负责人 |

> **注意**：这个外键是后来用 ALTER TABLE 加的，为什么？因为部门引用员工，员工又引用部门，形成了**循环引用**，必须先建好两张表，再加外键。

#### 4.3.1 为什么需要 ALTER TABLE 后加外键？

**生活中的类比**：你和你的朋友**互相写推荐信**
- 你给朋友写推荐信 → 需要朋友存在
- 朋友给你写推荐信 → 需要你存在
- **问题**：推荐信必须同时存在，但你俩谁先"存在"？

**回到数据库**：部门表和员工表就形成了这种"互相推荐"关系：

```
   department 部门表                    employee 员工表
   ┌────────────────┐                  ┌────────────────┐
   │ dept_code (主键)│                  │ emp_id (主键)  │
   │                │                  │                │
   │ dept_manager   │─────────────────▶│ (主键emp_id)   │
   │   _emp (外键)  │   负责人是谁      │                │
   │                │                  │ dept_code      │
   │                │◀─────────────────│ (外键) 属于哪个部门
   └────────────────┘                  └────────────────┘
        ↑                                  ↑
        │                                  │
        └────────── 互相依赖 ──────────────┘
```

**如果用 CREATE TABLE 同时建立两边会怎样？**

```sql
-- ❌ 错误示范：建部门表时试图引用还没建的员工表
CREATE TABLE department(
    ...
    FOREIGN KEY(dept_manager_emp) REFERENCES employee(emp_id)  -- 报错！
);
-- MySQL 报错：foreign key constraint is incorrectly formed
```

**解决方法：分两步走**

```sql
-- 第1步：建表时先不管"回头"的外键
CREATE TABLE department(
    dept_code VARCHAR(10) PRIMARY KEY,
    dept_manager_emp INT DEFAULT NULL  -- 字段先定义，不建外键
    -- 注意：没有 FOREIGN KEY
);

CREATE TABLE employee(
    emp_id INT PRIMARY KEY,
    dept_code VARCHAR(10),
    FOREIGN KEY(dept_code) REFERENCES department(dept_code)  -- ✅ OK
);

-- 第2步：表都建好了，再用 ALTER TABLE 补上"回头"的外键
ALTER TABLE department
ADD CONSTRAINT fk_dept_mgr
FOREIGN KEY(dept_manager_emp) REFERENCES employee(emp_id);  -- ✅ OK
```

#### 4.3.2 你SQL文件中的建表顺序

| 行号 | 操作 | 当时存在的表 | 当时不存在的表 | 能否成功？ |
|------|------|---------------|------------------|------------|
| **5-11** | 建 department 表 | （空） | position, employee | ✅ 没有外键，零依赖 |
| **14-21** | 建 position 表 | department | employee | ✅ 引用department，OK |
| **24-36** | 建 employee 表 | department, position | （无） | ✅ 引用department和position，OK |
| **39** | 给 department 加外键 | department, employee | （无） | ✅ employee已存在，可以加 |

> **核心原则**：**被引用的表必须先建好，引用的表后建**。遇到循环引用时，先把表都建好（不互加外键），最后再用 ALTER TABLE 把"回头"的外键补上。

## 五、参照完整性

**参照完整性**是数据库的一条重要规则：**外键的值必须在被引用的表中存在，或者为NULL**。

### 5.1 保证了什么？

| 操作 | 没有外键 | 有外键 |
|------|----------|--------|
| 插入一个不存在的部门代码 | ✅ 可以插入（数据错误！） | ❌ 被拒绝 |
| 删除还有员工引用的部门 | ✅ 可以删除（员工变成孤儿！） | ❌ 被拒绝 |
| 修改部门代码 | ✅ 可以改（员工引用失效！） | ✅ 跟着改（CASCADE） |

### 5.2 插入顺序很重要！

因为有外键约束，插入数据时必须**先插入被引用的表**：

```
正确的插入顺序：
1. 先插入 department（部门）     ← 没有外键依赖
2. 再插入 position（职位）       ← 依赖 department
3. 再插入 employee（员工）       ← 依赖 position 和 department
4. 再插入 payment_method（支付） ← 依赖 employee
5. 再插入 project（项目）        ← 没有外键依赖
6. 再插入 project_member        ← 依赖 employee 和 project
...
```

> **类比**：你得先有公司，才能有部门；先有部门，才能有员工。这是逻辑上的先后顺序。

## 六、外键的利与弊

| 优点 | 缺点 |
|------|------|
| 保证数据一致性 | 插入/删除要按顺序，稍微麻烦 |
| 防止脏数据 | 批量操作时可能影响性能 |
| 数据库层面自动检查 | 有些大型互联网项目选择不用外键，改用代码控制 |

> **实际开发中**：课程设计和中小项目**推荐用外键**；超大型互联网项目（如淘宝）为了性能，可能不用外键，而是在代码层面保证一致性。

## 七、本课小结

```
外键 FOREIGN KEY
├── 作用：保证引用的数据必须存在
├── 语法：FOREIGN KEY(本表字段) REFERENCES 另一张表(字段)
├── 删除策略
│   ├── RESTRICT  → 拒绝删除（最安全）
│   ├── CASCADE   → 跟着删除（依附关系强时用）
│   └── SET NULL  → 设为空（可选关系时用）
├── 更新策略
│   └── CASCADE   → 跟着修改（最常用）
└── 参照完整性
    └── 外键值必须在被引用表中存在，或为NULL
```

**核心记忆**：
1. 外键 = "查岗机制"，确保引用的数据真实存在
2. RESTRICT 最安全，CASCADE 跟着变，SET NULL 留空位
3. 插入数据要先插被引用的表，删除则反过来
