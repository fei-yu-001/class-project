# 第6课：索引 INDEX 的作用与设计

---

## 一、什么是索引？

想象你要在一本500页的书中找"数据库"这个词：

| 没有索引 | 有索引（目录） |
|----------|---------------|
| 从第1页翻到第500页，逐页找 | 翻到目录，看到"数据库……第127页"，直接翻到127页 |
| 慢！可能要翻500页 | 快！只翻2次 |

**索引就是数据库的"目录"**，让查询速度大幅提升。

---

## 二、索引的工作原理

### 2.1 没有索引时

```sql
SELECT * FROM employee WHERE emp_name = '张三';
```

数据库怎么做？→ **全表扫描**：从第1行开始，逐行检查 emp_name 是不是"张三"。

如果表有100万行，最坏情况要检查100万次。

### 2.2 有索引时

```sql
CREATE INDEX idx_emp_name ON employee(emp_name);
```

数据库怎么做？→ **索引查找**：在索引结构中快速定位"张三"在哪一行，直接跳过去。

就像查字典：有拼音目录，直接翻到Z开头的区域，不用从A开始翻。

## 三、你项目中的索引

```sql
-- 部门表
CREATE INDEX idx_dept_name ON department(dept_name);

-- 职位表
CREATE INDEX idx_pos_name ON position(pos_name);

-- 员工表
CREATE INDEX idx_emp_name ON employee(emp_name);
CREATE INDEX idx_emp_status ON employee(emp_status);

-- 工资记录表
CREATE INDEX idx_sal_period ON salary_record(pay_period);

-- 奖金表
CREATE INDEX idx_bonus_type ON bonus(bonus_type);

-- 扣款表
CREATE INDEX idx_deduct_type ON deduction(deduct_type);

-- 绩效表
CREATE INDEX idx_perf_emp ON performance(emp_id);

-- 考勤表
CREATE INDEX idx_att_emp ON attendance(emp_id);

-- 请假表
CREATE INDEX idx_leave_emp ON leave_req(emp_id);

-- 加班表
CREATE INDEX idx_ot_emp ON overtime(emp_id);

-- 职位变动表
CREATE INDEX idx_change_emp ON pos_change(emp_id);

-- 支付方式表
CREATE INDEX idx_pay_emp ON payment_method(emp_id);

-- 项目表
CREATE INDEX idx_proj_name ON project(proj_name);
```

### 3.1 索引分组解读

**第1组：常用查询字段索引（7个）**

这些字段经常出现在 `WHERE` 查询条件中：

| 索引名 | 索引字段 | 业务场景 |
|--------|----------|----------|
| idx_dept_name | department.dept_name | "查询所有技术部的员工" |
| idx_pos_name | position.pos_name | "查询所有Java工程师的薪资" |
| idx_emp_name | employee.emp_name | "查询叫张三的员工" |
| idx_emp_status | employee.emp_status | "查询所有在职员工" |
| idx_sal_period | salary_record.pay_period | "查询2024年1月的所有工资" |
| idx_bonus_type | bonus.bonus_type | "统计所有项目奖的总金额" |
| idx_deduct_type | deduction.deduct_type | "统计所有个人所得税总额" |
| idx_proj_name | project.proj_name | "查询'电商系统'项目的所有成员" |

**第2组：外键字段索引（6个）**

这些表经常通过 `emp_id` 与 `employee` 表做 `JOIN` 查询：

| 索引名 | 索引字段 | JOIN场景 |
|--------|----------|----------|
| idx_perf_emp | performance.emp_id | JOIN employee 查绩效历史 |
| idx_att_emp | attendance.emp_id | JOIN employee 查出勤情况 |
| idx_leave_emp | leave_req.emp_id | JOIN employee 查请假记录 |
| idx_ot_emp | overtime.emp_id | JOIN employee 查加班时长 |
| idx_change_emp | pos_change.emp_id | JOIN employee 查晋升/调岗历史 |
| idx_pay_emp | payment_method.emp_id | JOIN employee 查银行卡信息 |

> **为什么外键字段要加索引？** 因为 `JOIN` 时需要在外键字段上做匹配，如果没索引，数据库会对右表做全表扫描，表越大越慢。**项目里6张行为记录表都加了这种索引，是非常规范的做法**。

## 四、索引的分类

### 4.1 按功能分

| 类型 | 说明 | 例子 |
|------|------|------|
| **普通索引** | 最基本的索引，没有特殊限制 | `idx_emp_name` |
| **唯一索引** | 值不能重复 | 主键自动创建唯一索引 |
| **主键索引** | 特殊的唯一索引，每表只有一个 | `PRIMARY KEY(emp_id)` |

> **注意**：当你定义 PRIMARY KEY 时，数据库会**自动**创建主键索引，不需要手动创建。

### 4.2 按字段数量分

| 类型 | 说明 | 例子 |
|------|------|------|
| **单列索引** | 只对一个字段建索引 | `idx_emp_name ON employee(emp_name)` |
| **联合索引** | 对多个字段组合建索引 | `idx_emp_dept ON employee(dept_code, emp_status)` |

---

## 五、索引的命名规范

你项目中的命名方式：

```
idx_表名简写_字段名
```

| 索引名 | 含义 |
|--------|------|
| idx_emp_name | employee表的emp_name字段 |
| idx_sal_period | salary_record表的pay_period字段 |
| idx_perf_emp | performance表的emp_id字段 |

> **好习惯**：索引命名要有规律，一看就知道是哪个表的哪个字段。

## 六、什么时候该建索引？

### ✅ 适合建索引的场景

| 场景 | 原因 | 你项目中的例子 |
|------|------|---------------|
| 经常作为查询条件的字段 | 加速WHERE查找 | emp_name（按姓名查员工） |
| 经常用于JOIN的字段 | 加速表连接 | emp_id（员工关联考勤等） |
| 经常用于排序的字段 | 加速ORDER BY | pay_period（按周期排序工资） |
| 经常用于分组的字段 | 加速GROUP BY | emp_status（按状态分组统计） |

### ❌ 不适合建索引的场景

| 场景 | 原因 |
|------|------|
| 数据量很小的表 | 全表扫描也很快，索引反而浪费 |
| 频繁更新的字段 | 每次更新都要维护索引，拖慢写入速度 |
| 区分度很低的字段 | 如gender只有"男/女"，索引效果差 |

> **区分度**：一个字段有多少种不同的值。性别只有2种（区分度低），姓名有成千上万种（区分度高）。区分度越高，索引效果越好。

---

## 七、索引的代价

索引不是免费的午餐！

| 代价 | 说明 |
|------|------|
| **存储空间** | 每个索引都要占用额外的磁盘空间 |
| **写入变慢** | INSERT/UPDATE/DELETE 时，除了改数据，还要更新索引 |
| **维护成本** | 索引越多，维护越复杂 |

> **类比**：每加一个索引，就像多编一本目录。查书更快了，但每次改书的内容都要同时更新所有目录。

## 八、你项目中索引设计的分析

### 8.1 外键字段几乎都建了索引

```sql
CREATE INDEX idx_perf_emp ON performance(emp_id);
CREATE INDEX idx_att_emp ON attendance(emp_id);
CREATE INDEX idx_leave_emp ON leave_req(emp_id);
CREATE INDEX idx_ot_emp ON overtime(emp_id);
CREATE INDEX idx_change_emp ON pos_change(emp_id);
CREATE INDEX idx_pay_emp ON payment_method(emp_id);
```

**为什么？** 因为这些表经常通过 emp_id 和 employee 表做 JOIN 查询，索引能大幅加速连接。

### 8.2 常用查询字段建了索引

```sql
CREATE INDEX idx_emp_name ON employee(emp_name);    -- 按姓名搜索
CREATE INDEX idx_emp_status ON employee(emp_status); -- 按状态筛选
CREATE INDEX idx_sal_period ON salary_record(pay_period); -- 按周期查工资
```

**为什么？** 这些是业务中最常用的查询条件。

## 九、本课小结

```
索引 INDEX
├── 作用：像书的目录，加速查询
├── 类型
│   ├── 普通索引 → 最基本
│   ├── 唯一索引 → 值不能重复
│   └── 主键索引 → 自动创建
├── 适合建索引
│   ├── WHERE条件字段
│   ├── JOIN连接字段
│   ├── ORDER BY / GROUP BY字段
│   └── 区分度高的字段
├── 不适合建索引
│   ├── 数据量小的表
│   ├── 频繁更新的字段
│   └── 区分度低的字段
└── 代价
    ├── 占用存储空间
    └── 写入变慢
```

**核心记忆**：
1. 索引 = 目录，加速查询
2. 主键自动有索引，外键字段建议加索引
3. 索引不是越多越好，要权衡查询速度和写入代价
