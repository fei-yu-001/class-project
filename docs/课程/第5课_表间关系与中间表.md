# 第5课：表间关系（1:1、1:N、M:N）与中间表

---

## 一、为什么表之间要有关系？

如果把所有信息塞进一张表里：

| 工号 | 姓名 | 部门 | 职位 | 基本工资 | 项目1 | 项目2 | ... |
|------|------|------|------|----------|-------|-------|-----|
| 1001 | 张三 | 技术部 | 工程师 | 15000 | 项目A | 项目B | ... |

**问题**：
1. **数据重复**："技术部"出现了很多次，浪费空间
2. **更新麻烦**：技术部改名了，要改所有行
3. **难以扩展**：一个人参与了10个项目，要加10列？

所以要把数据**拆分到不同的表**，用关系连接起来。

---

## 二、三种关系类型

### 2.1 一对一（1:1）

**含义**：A表的一条记录，对应B表的一条且仅一条记录。

**你项目中的例子**：员工 ↔ 支付方式

```
员工表：一个员工
支付方式表：这个员工的唯一支付方式

一个员工只有一种支付方式
一种支付方式只属于一个员工
```

**怎么实现的？**

```sql
CREATE TABLE payment_method(
    emp_id INT NOT NULL COMMENT '员工工号',  -- 既是外键，又是主键！
    pay_type VARCHAR(20) NOT NULL COMMENT '支付方式类型',
    account_no VARCHAR(100) NOT NULL COMMENT '账号',
    card_type VARCHAR(20) DEFAULT NULL COMMENT '卡类型',
    PRIMARY KEY(emp_id),  -- emp_id做主键，保证一个员工只能有一条支付记录
    FOREIGN KEY(emp_id) REFERENCES employee(emp_id) ON DELETE CASCADE ON UPDATE CASCADE
);
```

**关键点**：`emp_id` 既是主键又是外键 → 一个员工只能有一条支付方式记录。

> **为什么不用一张表？** 有些信息不是每个人都有的，拆出去可以避免空值太多。而且支付信息可能需要特殊的安全处理，单独存放更方便管理。

---

### 2.2 一对多（1:N）—— 最常见

**含义**：A表的一条记录，对应B表的多条记录。

**你项目中的例子**：

#### 部门 ↔ 员工（一个部门有多个员工）

```
技术部（1个部门）
  ├── 张三（员工1）
  ├── 李四（员工2）
  └── 王五（员工3）
```

**怎么实现的？** 在"多"的那一方加外键：

```sql
CREATE TABLE employee(
    ...
    dept_code VARCHAR(10) NOT NULL COMMENT '所属部门代码',  -- 外键在员工表
    FOREIGN KEY(dept_code) REFERENCES department(dept_code)
    ...
);
```

> **记忆口诀**：外键永远加在"多"的那一方。

#### 你项目中所有的1:N关系

| "1"的一方 | "N"的一方 | 外键在哪 | 含义 |
|-----------|-----------|----------|------|
| department | employee | employee.dept_code | 一个部门有多名员工 |
| department | position | position.dept_code | 一个部门有多个职位 |
| position | employee | employee.pos_id | 一个职位有多名员工 |
| employee | attendance | attendance.emp_id | 一个员工有多条考勤 |
| employee | leave_req | leave_req.emp_id | 一个员工有多条请假 |
| employee | overtime | overtime.emp_id | 一个员工有多条加班 |
| employee | performance | performance.emp_id | 一个员工有多条考核 |
| employee | pos_change | pos_change.emp_id | 一个员工有多条变动记录 |
| employee | salary_record | salary_record.emp_id | 一个员工有多条工资记录 |

> **规律**：1:N 关系中，"N"的一方总是有一个外键指向"1"的一方。

### 2.3 多对多（M:N）

**含义**：A表的一条记录对应B表的多条，B表的一条记录也对应A表的多条。

**你项目中的例子**：

#### 员工 ↔ 项目

```
张三参与了：项目A、项目B、项目C     ← 一个员工参与多个项目
项目A的成员：张三、李四、王五       ← 一个项目有多个员工
```

**怎么实现的？** 必须用**中间表**！

```sql
CREATE TABLE project_member(
    emp_id INT NOT NULL COMMENT '员工工号',
    proj_id INT NOT NULL COMMENT '项目ID',
    role_name VARCHAR(30) DEFAULT NULL COMMENT '角色',        -- 中间表可以有自己的属性
    contrib_coeff DECIMAL(3,2) NOT NULL DEFAULT 1.00 COMMENT '贡献系数', -- 中间表自己的属性
    PRIMARY KEY(emp_id, proj_id),  -- 联合主键
    FOREIGN KEY(emp_id) REFERENCES employee(emp_id),
    FOREIGN KEY(proj_id) REFERENCES project(proj_id)
);
```

**中间表的结构**：

| emp_id | proj_id | role_name | contrib_coeff |
|--------|---------|-----------|---------------|
| 1001 | 2001 | 开发 | 1.00 |
| 1001 | 2002 | 测试 | 0.80 |
| 1002 | 2001 | 设计 | 1.20 |

> **中间表不只是连接**：它还可以有自己的属性（如角色、贡献系数），这些属性既不属于员工，也不属于项目，而是属于"员工参与项目"这个**关系**本身。

#### 5.3.1 contrib_coeff 的业务含义（重要思考）

`contrib_coeff`（贡献系数）究竟是什么意思？有两种理解：

**理解A：绝对系数（奖励倍数）** — 你SQL的默认理解
```
项目奖金 = 项目总奖金 × 自己的贡献系数
项目A奖金 10万：
  张三=1.0 → 拿 10万
  李四=0.5 → 拿 5万
  王五=1.5 → 拿 15万
每个人互相独立，加起来可以≠1
```

**理解B：占比系数（百分比）** — 你提出的理解
```
每个人占项目总贡献的比例
项目A成员：
  张三=0.5 → 50%
  李四=0.3 → 30%
  王五=0.2 → 20%
加起来必须=1（用 CHECK 约束保证）
```

> **你SQL用的是A方案**：`DEFAULT 1.00` 表示"标准贡献水平"，不要求加起来等于1。
>
> **如果想用B方案**，应该这样设计：
> ```sql
> contrib_ratio DECIMAL(3,2) NOT NULL  -- 不能默认，必须填
>   CHECK (contrib_ratio >= 0 AND contrib_ratio <= 1)
> ```

**设计经验**：字段名+默认值要让人一眼看明白含义。比如 `reward_factor`（奖励倍数）+ `DEFAULT 1.0` 就比 `contrib_coeff` + `DEFAULT 1.00` 更清晰。

#### 5.3.2 "分蛋糕"思维（重要的设计思想）

> **核心思想**：数据库设计的本质，是把现实世界的"分蛋糕"问题抽象成表结构。

**用你的项目实例来看"分蛋糕"思维**：

```
🍰 现实问题                            💾 数据库映射
─────────────────────────────────────────────────────
1. 公司有几个"蛋糕"？                  → project 项目表
   （项目的预算）

2. 每个蛋糕由哪些人分？                → project_member 中间表
   （谁参与了这个项目）

3. 每个人分多少？                      → contrib_coeff 贡献系数
   （分蛋糕的份额）
```

**一句话总结这个思维**：
> **项目 = 蛋糕，员工 = 分蛋糕的人，贡献系数 = 每个人该分多少。**

**这个思维能帮你举一反三**——以后遇到任何"分配"问题，都能用类似方式设计表：

| 现实问题 | 蛋糕 | 分蛋糕的人 | 份额 |
|----------|------|------------|------|
| 项目奖金分配 | 项目 | 项目成员 | 贡献系数 |
| 部门预算分配 | 部门预算 | 部门成员 | 分摊比例 |
| 班级座位分配 | 教室 | 学生 | 选座优先级 |
| 任务分配 | 任务 | 员工 | 工时占比 |

**这个思维的核心价值**：
1. **识别"蛋糕"** —— 找出要被分配的资源（项目/预算/任务/座位）
2. **识别"分蛋糕的人"** —— 找出参与分配的对象（员工/部门/学生）
3. **定义"分法"** —— 字段就是这个"分法"（系数/比例/优先级）

> **记住这个思维**：以后设计任何"分配类"业务表时，先问自己"蛋糕是什么？谁来分？怎么分？"，答案就出来了。

#### 你项目中所有的M:N关系

| 关系 | 中间表 | 中间表的额外属性 |
|------|--------|-----------------|
| 员工 ↔ 项目 | project_member | role_name（角色）、contrib_coeff（贡献系数） |
| 工资 ↔ 奖金 | sal_bonus_rel | 无 |
| 工资 ↔ 扣款 | sal_deduct_rel | 无 |

## 三、三种关系对比总结

| 关系 | 例子 | 实现方式 | 识别口诀 |
|------|------|----------|----------|
| 1:1 | 员工↔支付方式 | 外键+主键合一 | "一个萝卜一个坑" |
| 1:N | 部门↔员工 | 外键在N方 | "一母多子" |
| M:N | 员工↔项目 | 中间表 | "互相多对多" |

## 四、如何判断关系类型？

**问自己两个问题**：
1. A的一个，能对应B的几个？
2. B的一个，能对应A的几个？

| 答案1 | 答案2 | 关系 |
|-------|-------|------|
| 1个 | 1个 | 1:1 |
| 多个 | 1个 | 1:N（A是N方） |
| 多个 | 多个 | M:N |

**练习**：
- 一个班级有多少学生？多个。一个学生属于几个班级？1个。→ **1:N**
- 一个学生选几门课？多个。一门课有几个学生？多个。→ **M:N**
- 一个人有几个身份证？1个。一个身份证属于几个人？1个。→ **1:1**

### 5.3 关系图解读

**从图里你可以读出 3 个层次**：

**第1层：核心组织架构（1:N链条）**
```
部门 → 职位 → 员工
(1:N)  (1:N)
```
> 一个部门有多个职位，一个职位有多名员工

**第2层：员工的"行为记录"（1:N扇出）**
```
          员工 → 考勤、请假、加班、考核、职位变动、支付方式
```
> 员工有各种各样的"日常记录"，每种记录都是1:N

**第3层：M:N关系（需要中间表）**
```
员工 ↔ 项目
工资 ↔ 奖金
工资 ↔ 扣款
```
> 这3对关系是"多对多"，必须用中间表

## 六、本课小结

```
表间关系
├── 1:1（一对一）
│   ├── 实现：外键同时做主键
│   └── 例子：员工↔支付方式
├── 1:N（一对多）★最常见
│   ├── 实现：外键在"N"的一方
│   └── 例子：部门↔员工、职位↔员工
└── M:N（多对多）
    ├── 实现：必须用中间表
    ├── 中间表可以有额外属性
    └── 例子：员工↔项目、工资↔奖金
```

**核心记忆**：
1. 1:1 → 外键+主键合一
2. 1:N → 外键加在"多"的那一方
3. M:N → 必须建中间表，中间表可以有额外属性
