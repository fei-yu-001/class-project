# 第8课：存储引擎与字符集

---

## 一、什么是存储引擎？

你建表时每张表最后都有一句：

```sql
ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
```

**存储引擎**就是决定数据**怎么存、怎么取**的底层机制。

> **类比**：同样是运货，你可以用卡车、火车、飞机——都能运，但速度、成本、适用场景不同。存储引擎就是数据库的"运输工具"。

---

## 二、MySQL的两大存储引擎

### 2.1 InnoDB —— 你项目用的引擎

| 特性 | 说明 |
|------|------|
| **支持事务** | 一组操作要么全部成功，要么全部失败 |
| **支持外键** | 能保证数据引用完整性 |
| **行级锁** | 并发性能好，多人同时操作不冲突 |
| **崩溃恢复** | 意外断电后数据不丢失 |

### 2.2 MyISAM —— 老牌引擎

| 特性 | 说明 |
|------|------|
| **不支持事务** | 操作不能回滚 |
| **不支持外键** | 无法保证引用完整性 |
| **表级锁** | 并发性能差，一个人写整张表都锁住 |
| **速度快** | 读操作比InnoDB快（只读场景） |

### 2.3 对比

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | ✅ 支持 | ❌ 不支持 |
| 外键 | ✅ 支持 | ❌ 不支持 |
| 锁粒度 | 行级锁（细） | 表级锁（粗） |
| 崩溃恢复 | ✅ 自动恢复 | ❌ 可能损坏 |
| 全文索引 | ✅ 支持（5.6+） | ✅ 支持 |
| 适用场景 | 大多数场景 | 只读/统计报表 |

> **结论**：**InnoDB 是默认选择**，只有极少数纯读场景才考虑 MyISAM。

## 三、事务（Transaction）—— InnoDB 的核心

### 3.1 什么是事务？

事务是一组操作，**要么全部成功，要么全部失败**。

**生活中的例子**：银行转账
```
张三给李四转1000元：
1. 张三账户 -1000 ✅
2. 李四账户 +1000 ❌（系统故障）

如果没有事务：张三少了1000，李四没收到 → 钱凭空消失了！
如果有事务：第2步失败，第1步自动撤销 → 两边都没变，安全！
```

### 3.2 事务的ACID特性

| 特性 | 英文 | 含义 | 举例 |
|------|------|------|------|
| **原子性** | Atomicity | 全部成功或全部失败 | 转账要么完成，要么不转 |
| **一致性** | Consistency | 数据从一个正确状态变到另一个正确状态 | 转账前后总金额不变 |
| **隔离性** | Isolation | 多个事务互不干扰 | 两人同时转账不冲突 |
| **持久性** | Durability | 提交后数据永久保存 | 断电也不丢数据 |

### 3.3 事务的SQL

```sql
START TRANSACTION;           -- 开始事务

UPDATE account SET balance = balance - 1000 WHERE name = '张三';
UPDATE account SET balance = balance + 1000 WHERE name = '李四';

COMMIT;                      -- 全部成功，提交！
-- 或者
ROLLBACK;                    -- 出错了，回滚！撤销所有操作
```

### 3.4 你项目中的事务场景

发工资就是一个典型的事务：

```
1. 计算应发工资
2. 计算奖金
3. 计算扣款
4. 生成工资记录
5. 关联奖金和扣款

如果第4步失败了，前面1-3步的计算也要撤销 → 需要事务！
```

## 四、字符集详解

### 4.1 什么是字符集？

计算机只认识0和1，字符集就是**文字和数字之间的翻译规则**。

| 字符 | ASCII编码 | UTF-8编码 |
|------|-----------|-----------|
| A | 65 | 65 |
| 中 | ❌ 不支持 | E4 B8 AD |
| 😀 | ❌ 不支持 | F0 9F 98 80 |

### 4.2 MySQL中的字符集层级

```
服务器级别 → 数据库级别 → 表级别 → 字段级别
```

每一级都可以单独设置，**子级会继承父级的设置**。

你的项目：
```sql
-- 数据库级别
CREATE DATABASE ... DEFAULT CHARACTER SET utf8mb4 ...

-- 表级别
CREATE TABLE ... ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 ...

-- 字段级别（一般不单独设，继承表的设置）
```

### 4.3 utf8 vs utf8mb4

| 对比 | utf8 | utf8mb4 |
|------|------|---------|
| 每字符最多字节 | 3字节 | 4字节 |
| 中文 | ✅ | ✅ |
| emoji表情 | ❌ | ✅ |
| 罕见汉字 | ❌ 部分 | ✅ |
| 是真UTF-8？ | ❌ MySQL的"残缺版" | ✅ 真正的UTF-8 |

> **铁律**：在MySQL中，**永远用 utf8mb4**，不要用 utf8。MySQL的 utf8 最多3字节，不是真正的UTF-8标准。

## 五、排序规则（Collation）

排序规则决定**怎么比较和排序文字**。

### 5.1 你项目用的 utf8mb4_general_ci

```sql
COLLATE utf8mb4_general_ci
```

| 部分 | 含义 |
|------|------|
| utf8mb4 | 字符集 |
| general | 通用排序算法 |
| ci | Case Insensitive（不区分大小写） |

### 5.2 ci vs cs vs bin

| 排序规则 | 大小写 | 举例 |
|----------|--------|------|
| `general_ci` | 不区分 | 'ABC' = 'abc' |
| `general_cs` | 区分 | 'ABC' ≠ 'abc' |
| `bin` | 按二进制比较 | 'ABC' ≠ 'abc'，速度最快 |

### 5.3 实际影响

```sql
-- 如果用 general_ci
SELECT * FROM employee WHERE emp_name = 'zhangsan';
-- 能匹配到 'ZhangSan'、'ZHANGSAN' 等

-- 如果用 bin
SELECT * FROM employee WHERE emp_name = 'zhangsan';
-- 只能匹配到 'zhangsan'，大小写必须完全一致
```

> **大多数场景用 general_ci**，除非你有明确的大小写区分需求（如用户名登录）。

## 六、你项目中的完整建表参数

```sql
CREATE TABLE employee(
    ...
)ENGINE=InnoDB AUTO_INCREMENT=1001 DEFAULT CHARSET=utf8mb4 COMMENT='员工表';
```

| 参数 | 值 | 含义 |
|------|-----|------|
| ENGINE | InnoDB | 使用InnoDB引擎，支持事务和外键 |
| AUTO_INCREMENT | 1001 | 自增从1001开始 |
| DEFAULT CHARSET | utf8mb4 | 字符集为utf8mb4，支持中文和emoji |
| COMMENT | '员工表' | 表注释 |

## 七、本课小结

```
存储引擎
├── InnoDB（推荐）→ 支持事务、外键、行级锁、崩溃恢复
├── MyISAM → 不支持事务和外键，只适合纯读场景
└── 事务 ACID
    ├── 原子性 → 全成功或全失败
    ├── 一致性 → 状态正确转换
    ├── 隔离性 → 互不干扰
    └── 持久性 → 永久保存

字符集
├── utf8mb4 → 永远用这个！支持中文+emoji+罕见字
├── utf8 → MySQL的残缺版，不要用
└── 排序规则
    ├── general_ci → 不区分大小写（推荐）
    └── bin → 区分大小写，按二进制比较
```

**核心记忆**：
1. InnoDB 是默认引擎，支持事务和外键
2. 事务 = 要么全成功，要么全失败（ACID）
3. 字符集永远用 utf8mb4，排序规则一般用 general_ci
