# 第7课：视图 VIEW 的概念与用法

---

## 一、什么是视图？

想象你经常需要查"员工的全信息"（姓名+部门+职位+工资），每次都要写这么长的SQL：

```sql
SELECT e.emp_id, e.emp_name, e.gender, e.birth_date, e.hire_date, e.emp_status,
       p.pos_id, p.pos_name, p.base_salary,
       d.dept_code, d.dept_name
FROM employee e
JOIN position p ON e.pos_id = p.pos_id
JOIN department d ON e.dept_code = d.dept_code;
```

太长了！如果能把这段查询**起个名字**，以后直接用这个名字查就好了。

**视图就是这个"名字"**——它是一段查询的快捷方式，看起来像表，但实际上不存数据。

> **通俗理解**：视图 = 保存的查询。就像你把一段常用的搜索条件保存下来，下次一键就能用。

---

## 二、视图 vs 表

| 对比 | 表（Table） | 视图（View） |
|------|-------------|-------------|
| 存数据吗？ | ✅ 存真实数据 | ❌ 不存数据，只存查询逻辑 |
| 占空间吗？ | ✅ 占磁盘空间 | ❌ 几乎不占空间 |
| 更新数据？ | ✅ 直接更新 | ⚠️ 部分可以，部分不行 |
| 速度？ | 直接读，快 | 每次查询时执行底层SQL，稍慢 |

> **类比**：表是真实的文件柜，视图是窗户——透过窗户看到的是文件柜里的东西，但窗户本身不存文件。

## 三、你项目中的5个视图

### 3.1 视图1：员工全信息

```sql
CREATE VIEW v_emp_full AS
SELECT e.emp_id, e.emp_name, e.gender, e.birth_date, e.hire_date, e.emp_status,
       p.pos_id, p.pos_name, p.base_salary,
       d.dept_code, d.dept_name
FROM employee e
JOIN position p ON e.pos_id = p.pos_id
JOIN department d ON e.dept_code = d.dept_code;
```

**作用**：把员工、职位、部门三张表的信息合在一起，查员工全信息只需：

```sql
SELECT * FROM v_emp_full WHERE emp_name = '张三';
```

**涉及知识点**：
- `JOIN`：内连接，只返回三张表都能匹配上的记录
- `e.emp_id` 中的 `e` 是 employee 表的别名

---

### 3.2 视图2：工资明细

```sql
CREATE VIEW v_sal_detail AS
SELECT s.salary_id, s.pay_period, s.gross_total, s.deduct_total, s.net_pay,
       e.emp_id, e.emp_name,
       b.bonus_id, b.bonus_type, b.pre_tax_amt,
       d.deduct_id, d.deduct_type, d.deduct_amt
FROM salary_record s
JOIN employee e ON s.emp_id = e.emp_id
LEFT JOIN sal_bonus_rel sb ON s.salary_id = sb.salary_id
LEFT JOIN bonus b ON sb.bonus_id = b.bonus_id
LEFT JOIN sal_deduct_rel sd ON s.salary_id = sd.salary_id
LEFT JOIN deduction d ON sd.deduct_id = d.deduct_id;
```

**作用**：查某月的工资条，包含奖金和扣款明细。

**新知识点：LEFT JOIN（左连接）**

| 连接类型 | 含义 | 举例 |
|----------|------|------|
| JOIN（内连接） | 两边都要有匹配才返回 | 有奖金才显示 |
| LEFT JOIN（左连接） | 左边表全返回，右边没匹配填NULL | 没奖金也显示，奖金列为NULL |

**为什么用 LEFT JOIN？** 因为不是每条工资都有奖金或扣款，用内连接会导致没有奖金的工资记录消失！

```
JOIN 的效果：
工资记录1 + 奖金A → 显示 ✅
工资记录2（无奖金）→ 消失 ❌

LEFT JOIN 的效果：
工资记录1 + 奖金A → 显示 ✅
工资记录2 + NULL  → 显示 ✅（奖金列为空）
```

### 3.3 视图3：员工考勤绩效

```sql
CREATE VIEW v_emp_perf_att AS
SELECT e.emp_id, e.emp_name, pr.grade, a.att_status
FROM employee e
LEFT JOIN performance pr ON e.emp_id = pr.emp_id
LEFT JOIN attendance a ON e.emp_id = a.att_id;
```

**作用**：同时查看员工的绩效等级和考勤状态。

**注意**：这里 `a.att_id` 的连接条件可能有问题（应该用 `a.emp_id`），这是一个潜在的bug。正确的写法应该是：

```sql
LEFT JOIN attendance a ON e.emp_id = a.emp_id
```

> **学习点**：写SQL时一定要仔细检查连接条件，用错字段会导致数据错乱。

---

### 3.4 视图4：员工项目参与

```sql
CREATE VIEW v_emp_project AS
SELECT e.emp_id, e.emp_name, p.proj_id, p.proj_name, pm.role_name, pm.contrib_coeff
FROM employee e
JOIN project_member pm ON e.emp_id = pm.emp_id
JOIN project p ON pm.proj_id = p.proj_id;
```

**作用**：查看员工参与了哪些项目，以及角色和贡献系数。

**这里用 JOIN 而不是 LEFT JOIN**：因为这个视图只关心"参与了项目的员工"，没参与项目的员工不需要显示。

---

### 3.5 视图5：员工薪资结算账户

```sql
CREATE VIEW v_emp_payinfo AS
SELECT e.emp_id, e.emp_name, pm.pay_type, pm.account_no, pm.card_type
FROM employee e
JOIN payment_method pm ON e.emp_id = pm.emp_id;
```

**作用**：查看员工的支付方式信息，发工资时用。

## 四、JOIN 类型总结

这是数据库查询中最重要的知识点之一！

### 4.1 图解对比

假设有两张表：
- 员工表：张三、李四、王五
- 考勤表：张三（出勤）、李四（迟到）、赵六（出勤）

```
JOIN（内连接）：两边都有的才显示
  → 张三、李四

LEFT JOIN（左连接）：左边全显示，右边没有填NULL
  → 张三、李四、王五（考勤为NULL）

RIGHT JOIN（右连接）：右边全显示，左边没有填NULL
  → 张三、李四、赵六（员工为NULL）
```

### 4.2 选择指南

| 需求 | 用什么 |
|------|--------|
| 两边都要有数据 | JOIN |
| 左边全要，右边可有可无 | LEFT JOIN |
| 右边全要，左边可有可无 | RIGHT JOIN（很少用） |

> **实际开发中**：90%的情况用 JOIN 或 LEFT JOIN，RIGHT JOIN 极少使用。

## 五、视图的优缺点

### 优点

| 优点 | 说明 |
|------|------|
| **简化查询** | 复杂SQL保存为视图，以后一行搞定 |
| **数据安全** | 只暴露需要的列，隐藏敏感信息 |
| **统一接口** | 多个应用用同一个视图，底层表变了视图可以不变 |

### 缺点

| 缺点 | 说明 |
|------|------|
| **性能** | 每次查视图都要执行底层SQL |
| **更新限制** | 复杂视图不能直接INSERT/UPDATE |
| **调试困难** | 视图套视图，出问题不好排查 |

## 六、视图的实际使用

```sql
-- 创建视图
CREATE VIEW v_emp_full AS SELECT ...;

-- 使用视图（和查表一模一样）
SELECT * FROM v_emp_full;

-- 带条件查询
SELECT * FROM v_emp_full WHERE dept_name = '技术部';

-- 删除视图
DROP VIEW v_emp_full;

-- 修改视图（先删再建，或用 CREATE OR REPLACE）
CREATE OR REPLACE VIEW v_emp_full AS SELECT ...;
```

## 七、本课小结

```
视图 VIEW
├── 本质：保存的查询，虚拟的表
├── 不存数据，每次查询时动态执行
├── 作用
│   ├── 简化复杂查询
│   ├── 隐藏敏感数据
│   └── 提供统一数据接口
├── JOIN 类型
│   ├── JOIN（内连接）→ 两边都要匹配
│   ├── LEFT JOIN（左连接）→ 左边全保留
│   └── RIGHT JOIN（右连接）→ 右边全保留
└── 使用
    ├── CREATE VIEW ... AS SELECT ...
    ├── SELECT * FROM 视图名
    └── DROP VIEW 视图名
```

**核心记忆**：
1. 视图 = 保存的查询快捷方式，不存数据
2. JOIN 只返回匹配的，LEFT JOIN 左边全保留
3. 视图简化了查询，但不能替代表的存储功能
