# 第9课：ER图与数据库设计思路总结

---

## 一、什么是ER图？

ER图（Entity-Relationship Diagram）= 实体-关系图，是数据库设计的**蓝图**。

> **类比**：盖楼之前先画建筑图纸，建数据库之前先画ER图。

---

## 二、ER图的三大要素

| 要素 | 符号 | 含义 | 你项目中的例子 |
|------|------|------|---------------|
| **实体（Entity）** | 矩形 | 现实中的"东西" | 员工、部门、项目 |
| **属性（Attribute）** | 椭圆 | 实体的特征 | 姓名、工号、工资 |
| **关系（Relationship）** | 菱形 | 实体之间的联系 | 属于、参与、包含 |

### 2.1 实体 → 表

```
[员工]  →  employee 表
[部门]  →  department 表
[项目]  →  project 表
```

### 2.2 属性 → 字段

```
员工实体的属性：
  (工号)  →  emp_id
  (姓名)  →  emp_name
  (性别)  →  gender
```

带下划线的属性 = 主键，如 <u>工号</u>

### 2.3 关系 → 外键 / 中间表

```
[员工] ──属于── <部门>    →  employee表的dept_code外键
[员工] ──参与── <项目>    →  project_member中间表
```

## 四、数据库设计步骤

从你的项目可以看出，设计一个数据库通常遵循以下步骤：

### 第1步：需求分析

**要管理什么？**
- 公司有部门和职位
- 员工有基本信息、考勤、绩效
- 员工参与项目
- 要计算工资（基本工资+奖金-扣款）

### 第2步：识别实体

从需求中提取"名词"：

| 需求描述 | 实体 |
|----------|------|
| 公司有**部门**和**职位** | 部门、职位 |
| **员工**有基本信息 | 员工 |
| 员工参与**项目** | 项目 |
| 要计算**工资** | 工资记录 |
| **奖金**和**扣款** | 奖金、扣款 |

### 第3步：识别关系

从需求中提取"动词"：

| 需求描述 | 关系 | 类型 |
|----------|------|------|
| 部门有多个职位 | 属于 | 1:N |
| 员工属于一个部门 | 属于 | 1:N |
| 员工担任一个职位 | 担任 | 1:N |
| 员工参与多个项目 | 参与 | M:N |
| 工资包含多个奖金 | 包含 | M:N |
| 工资扣除多个扣款 | 扣除 | M:N |
| 员工有一种支付方式 | 绑定 | 1:1 |

### 第4步：确定属性

为每个实体确定字段：

| 实体 | 属性 |
|------|------|
| 员工 | 工号(PK)、姓名、性别、出生日期、入职日期、在职状态、职位ID(FK)、部门代码(FK) |
| 部门 | 部门代码(PK)、部门名称、上级部门代码、负责人工号(FK) |

### 第5步：建表并加约束

```sql
CREATE TABLE ... (
    字段 数据类型 约束,
    PRIMARY KEY(...),
    FOREIGN KEY(...) REFERENCES ...
);
```

### 第6步：加索引和视图

- 为常用查询字段加索引
- 为复杂查询创建视图

## 五、数据库设计原则

### 5.1 三大范式

| 范式 | 核心要求 | 你的项目中的体现 |
|------|----------|-----------------|
| **第一范式（1NF）** | 每个字段不可再分 | 没有把"姓名"拆成"姓"+"名"放在一个字段里 |
| **第二范式（2NF）** | 非主键字段完全依赖主键 | project_member表的角色和贡献系数依赖(emp_id, proj_id)联合主键 |
| **第三范式（3NF）** | 非主键字段不互相依赖 | 员工表不存"基本工资"，因为工资属于职位，通过pos_id关联获取 |

**3NF的例子**：

```
❌ 违反3NF的做法（员工表直接存基本工资）：
employee: emp_id, emp_name, base_salary, pos_id
问题：如果职位的基本工资变了，要更新所有该职位的员工 → 容易遗漏

✅ 符合3NF的做法（基本工资存在职位表）：
position: pos_id, pos_name, base_salary
employee: emp_id, emp_name, pos_id
需要工资时，通过pos_id关联查询
```

### 5.2 其他设计原则

| 原则 | 说明 | 例子 |
|------|------|------|
| **主键要稳定** | 主键不要用会变的值 | 用emp_id而不是emp_name |
| **适当冗余** | 有时为了性能可以适度冗余 | salary_record中的base_snap（基本工资快照） |
| **命名规范** | 表名、字段名要有规律 | dept_code、emp_id、proj_id |

> **关于base_snap（工资快照）**：salary_record表存了 base_snap（基本工资快照），虽然职位表已经有base_salary，但工资记录需要保存**发工资那一刻**的基本工资，因为职位工资可能后来变了。这是**合理冗余**。

## 六、全部9课知识串联

```
数据库设计流程：

1. 需求分析 → 确定要管理什么
       ↓
2. 画ER图 → 识别实体、属性、关系
       ↓
3. 建库 → CREATE DATABASE (第1课)
       ↓
4. 建表 → CREATE TABLE (第1课)
   ├── 选择数据类型 (第2课)
   ├── 设置字段约束 (第2课)
   ├── 定义主键 (第3课)
   ├── 定义外键 (第4课)
   └── 确定表间关系 (第5课)
       ↓
5. 加索引 → CREATE INDEX (第6课)
       ↓
6. 建视图 → CREATE VIEW (第7课)
       ↓
7. 选择存储引擎和字符集 (第8课)
       ↓
8. 遵循设计原则和范式 (第9课)
```

## 七、你的项目完整知识地图

```
salary_management 数据库
│
├── 基础设施
│   ├── 字符集：utf8mb4（支持中文+emoji）
│   ├── 排序规则：utf8mb4_general_ci（不区分大小写）
│   └── 存储引擎：InnoDB（支持事务+外键）
│
├── 14张表
│   ├── 核心表：department, position, employee
│   ├── 项目管理：project, project_member
│   ├── 考勤绩效：attendance, leave_req, overtime, performance
│   ├── 薪资核算：salary_record, bonus, deduction
│   ├── 关联表：sal_bonus_rel, sal_deduct_rel
│   └── 支付信息：payment_method
│
├── 关系
│   ├── 1:1 → employee ↔ payment_method
│   ├── 1:N → department → employee, position → employee 等
│   └── M:N → employee ↔ project, salary ↔ bonus, salary ↔ deduction
│
├── 14个索引
│   ├── 外键字段索引（加速JOIN）
│   └── 常用查询字段索引（加速WHERE）
│
└── 5个视图
    ├── v_emp_full（员工全信息）
    ├── v_sal_detail（工资明细）
    ├── v_emp_perf_att（考勤绩效）
    ├── v_emp_project（项目参与）
    └── v_emp_payinfo（支付信息）
```
