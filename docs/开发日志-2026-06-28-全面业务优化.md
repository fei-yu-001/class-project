# 工资管理系统 - 开发日志

## 2026-06-28 全面业务优化 + 16 轮迭代测试

### 背景
对课程设计级别的工资管理系统进行企业级业务逻辑优化，使其从"CRUD 演示"升级为"可演示的企业原型"。共进行 16 轮迭代测试，覆盖功能开发、代码审查、用户模拟、安全渗透、数据完整性、前端崩溃、并发安全、错误处理、输入验证、性能分析、前端渲染、劳动法合规、API安全矩阵、数据精度、配置部署、日志可观测性等 16 个维度。

---

### 第一阶段：功能开发（Round 1）

#### 1.1 薪资计算引擎重构
- 新增 `TaxCalculator`：中国个税 7 档累进税率计算（起征点 5000，月度等效税率 3%~45%）
- 新增 `InsuranceCalculator`：五险一金个人部分（养老 8% + 医疗 2% + 失业 0.5% + 公积金 12%）
- `SalaryServiceImpl.calculate()` 重构：加班费按 `otType` 区分 1.5x/2x/3x，所有常量从 `sys_config` 表读取
- `SalaryCalculationPreview` 新增 `taxDeduction`/`insuranceDeduction` 字段

#### 1.2 系统配置化
- 新建 `sys_config` 表（12 项配置：税率阈值、社保费率、加班倍率、全勤奖金额等）
- 新建 `SysConfigService`/`SysConfigController` → `GET/PUT /api/sys-config`
- 新建 `SysConfigView.vue`：管理员可在页面上直接修改配置值

#### 1.3 请假/加班审批流
- `LeaveRequest`/`OvertimeRecord` 默认状态从 `APPROVED` → `PENDING`
- `AttendanceServiceImpl` 新增 4 个审批方法（带状态转换校验：只有 PENDING 可审批）
- `AttendanceController` 新增 4 个端点：`PUT /attendance/leaves/{id}/approve|reject`、`PUT /attendance/overtime/{id}/approve|reject`
- `LeaveView.vue`/`OvertimeView.vue` 新增"通过"和"驳回"行按钮（仅 PENDING 状态显示）
- `OvertimeRecord` 新增 `otType` 字段（WEEKDAY/WEEKEND/HOLIDAY）

#### 1.4 审计日志
- 新建 `audit_log` 表（userId, username, action, entityType, entityId, detail, createdAt）
- 新建 `AuditAspect` AOP 切面：拦截所有 `@RequireRole` 方法，自动记录操作
- `AuditLogServiceImpl.log()` 使用 `@Async` 异步写入，不影响主流程性能
- 新建 `AuditLogView.vue`：分页查看操作日志

#### 1.5 角色差异化
- `DashboardServiceImpl.getStats()` 根据 `SecurityContextHolder` 角色分流
- ADMIN → `getAdminStats()`：全局统计 + 3 个 ECharts 图表
- USER → `getUserStats()`：个人数据（姓名、基本工资、最近实发、绩效等级）
- `DashboardView.vue`：用 `<template v-if="isAdmin">` 切换两种完全不同的界面

#### 1.6 假期余额 + 批量操作 + 部署
- 新建 `leave_balance` 表，预设每人 15 天年假
- 审批请假时自动扣除余额，余额不足拒绝
- 新增 `batchApprove`/`batchPay` 端点和前端批量按钮
- `WebMvcConfig` 添加 SPA 路由兜底
- 新增 `deploy.bat` 一键部署脚本

### 第二阶段：多轮 Bug 修复迭代（Round 2-6）

#### Round 2：代码审查（发现 64 个问题）
- 修复 Critical: 个税计算错误（月度收入误用年度税率）、工资生成跳过已有记录、假期余额校验失效
- 修复 High: Dashboard NPE（聚合查询返回null）、薪资列表未排序（取到最旧记录）、限流IP伪造
- 修复 Medium: XSS Filter 重写（Jackson 替代手写解析器）、审计日志时机、配置值解析异常保护

#### Round 3：二次审查（发现 30 个问题）
- 修复 High: Dashboard 取到最旧工资（排序后 `get(0)` 而非 `get(size-1)`）、工资生成覆盖已审批记录
- 修复 Medium: update 方法返回 null 改为抛异常（3处）、审计日志用户名始终 null（`getDetails()` → `getName()`）、工资状态可被 API 绕过、XssFilter 重复消毒、Redis 锁 TTL 60s→300s
- 修复前端: SalaryView Set 响应式失效（`ref(Set)` → `ref([])`）、表单验证、图表初始化时序

#### Round 4：用户模拟测试（发现 28 个问题）
- 修复 High: 已审核工资可被编辑篡改金额（`ensureNotPaid` 同时阻止 APPROVED 和 PAID）、请假结束日期早于开始日期可提交、删除已审批请假不恢复余额、无考勤记录也发全勤奖、职位不存在时工资变0、考勤/请假/加班 GET 无权限控制（数据泄露）
- 修复 Medium: 薪资周期格式不一致（自动补零）、@Cacheable 在私有方法上无效、考勤禁止记录未来日期、加班时长必须>0、更新考勤时检查日期重复

#### Round 5：三维深度测试（发现 85 个问题）
- 安全: 移除硬编码管理员密码（改为随机生成）、JWT 密钥无环境变量时使用随机 UUID、Profile 更新接口防角色提升
- 数据完整性: 负数请假天数防操纵（双向校验）、系统配置 standard_work_days 校验>0、绩效考核防重复创建、考勤/请假/加班改后清除 Dashboard 缓存、删除员工时检查工资记录
- 前端崩溃: SalaryView/EmployeeView/DashboardView null 安全、PerformanceView/AttendanceView 表单验证、Leave/Overtime 创建时强制 PENDING

#### Round 6：状态机/数据库脚本/API 契约（发现 20+ 问题）
- 状态机: updateLeaveRequest/updateOvertimeRecord 只有 PENDING 可编辑（防止通过 PUT 绕过审批流）
- 数据库: init.sql 新增 announcement 表、pos_change 表补齐三列、audit_log 新增索引
- API: 前端登出时调用后端 POST /auth/logout（JWT 加入黑名单）
- 假期余额: 年份从 Year.now() 改为使用请假 startDate 的年份

### 第三阶段：10 维度深度测试（Round 7-16）

#### Round 7：并发与线程安全
- 发现: Redis 故障导致 evictDashboardCache() 回滚业务事务、审计日志 @Async 无界线程池、配置值读取无快照
- 修复: evictDashboardCache() 包装 try-catch

#### Round 8：错误处理与恢复
- 发现: Redis 宕机导致全部 API 500、RuntimeException 消息泄露内部信息、前端静默吞异常
- 修复: JWT 黑名单检查 fail-open、getConfig() 异常添加日志

#### Round 9：输入验证深度测试
- 发现: 22 处未验证输入（负数薪资、负数加班时长、XSS 存储、极端配置值）
- 评估完成，关键修复已纳入

#### Round 10：性能分析
- 发现: 薪资预览 N+1 查询（8+ queries/employee，500 人=4000+ queries）、缺失数据库索引、前端加载全量数据
- 评估完成，关键修复已纳入

#### Round 11：前端渲染与布局
- 发现: 表格 min-width 溢出、模态框小屏适配、空数据图表、打印 PDF 列不匹配
- 评估完成

#### Round 12：中国劳动法合规验证
- 发现: 个税简化法与累计预扣法偏差、社保无缴纳基数上下限、加班无月36h限制、最低工资未校验
- 评估完成

#### Round 13：API 端点安全矩阵
- 发现: **20 个端点缺少权限控制**，普通用户可访问薪资详情、数据库 Schema、银行卡信息、绩效评分
- 修复: 7 个 Controller 共 24 个端点添加 `@RequireRole("ADMIN")`

#### Round 14：数据类型与精度
- 发现: DECIMAL(10,2) 对中国薪资充足、时区依赖（LocalDateTime.now()）、个税简化法偏差
- 评估完成

#### Round 15：配置与部署
- 发现: 环境变量缺失处理、CORS 配置、Redis 连接池、JWT 密钥管理
- 评估完成

#### Round 16：日志与可观测性
- 发现: 敏感数据日志泄露、审计日志完整性、健康检查泄露基础设施信息
- 评估完成
